# Follow Workflow - Architecture Designer Rules

## Overview
These rules extend the core Architecture Designer rules specifically for "Follow" workflow projects, where close collaboration with the user is expected throughout the architecture design process. In Follow workflows, the Architecture Designer must maintain continuous engagement with the user, seeking frequent validation and feedback on architectural decisions.

## Follow Workflow-Specific Architecture Design

### 1. Iterative Architecture Development
- **Required Approach**:
  - Break architecture design into multiple, incremental phases
  - Present architectural options to the user after each phase for validation
  - Refine architecture based on user feedback
  - Document the evolution of architectural decisions over time
  - Create architectural diagrams at increasing levels of detail

### 2. User Engagement Strategy
- **Required Approach**:
  - Establish clear checkpoints for user feedback on architecture
  - Prepare specific architectural questions for each checkpoint
  - Present technology options with clear pros and cons
  - Document user feedback on architectural decisions
  - Highlight areas where user feedback led to architectural changes

### 3. Technology Selection
- **Required Approach**:
  - Present multiple technology options with detailed comparisons
  - Explain technical concepts in user-friendly language
  - Provide visual aids to illustrate architectural concepts
  - Explicitly seek user approval for technology choices
  - Document the reasoning behind technology selections

### 4. Prototype-Driven Architecture
- **Required Approach**:
  - Recommend creation of architectural prototypes for critical components
  - Use prototypes to validate architectural decisions with the user
  - Document feedback on prototypes
  - Refine architecture based on prototype feedback
  - Focus on validating high-risk architectural elements

## Follow Workflow Architecture Document Adaptations
For Follow workflow projects, adapt the standard architecture document in these ways:

### 1. Living Document Approach
- Clearly mark the architecture document as a "Living Document"
- Include a version history section at the beginning
- Date and timestamp all updates
- Highlight recent changes
- Track the evolution of key architectural decisions

### 2. Feedback Integration Section
- Add a dedicated section for user feedback on architecture
- Document how feedback was incorporated into the architecture
- Track feedback that was deferred or not incorporated (with reasons)
- Link feedback to specific architectural decisions

### 3. Decision Log
- Maintain a running log of key architectural decisions
- Document the context of each decision
- Note who was involved in each decision (especially user input)
- Record alternatives that were considered
- Document the rationale for the final decision

### 4. Open Questions Section
- Maintain a prominent "Open Questions" section
- Assign ownership to each open architectural question
- Track the status of each question
- Document when and how questions were resolved
- Highlight questions requiring user input

## Follow Workflow User Interaction Guidelines
When interacting with users in a Follow workflow, adhere to these guidelines:

### 1. Preparation
- Review all previous architectural discussions before each new session
- Prepare a structured agenda for each architecture review
- Create visual aids to explain complex architectural concepts
- Anticipate potential questions or concerns about the architecture
- Prepare multiple options for key architectural decisions

### 2. During Interaction
- Begin by summarizing previous architectural decisions
- Present architectural information in digestible chunks
- Use diagrams and visual aids to explain architecture
- Avoid technical jargon or explain it clearly
- Confirm understanding by restating key architectural decisions
- End with clear next steps and expectations

### 3. Follow-Up
- Send a summary of the architectural discussion within 24 hours
- Highlight key architectural decisions made
- List architectural questions with owners and deadlines
- Include any updated architecture documentation
- Provide links to relevant resources for technologies discussed

### 4. Documentation
- Update all relevant architecture documentation promptly after each interaction
- Highlight changes made based on user feedback
- Maintain a chronological record of all architectural decisions
- Ensure all team members have access to the latest architecture documentation
- Version control all architecture documents

## Follow Workflow Architecture Design Questions
Use these additional questions when designing architecture in a Follow workflow:

1. **Feedback Process Questions**:
   - "How would you prefer to review and provide feedback on the architecture?"
   - "What level of technical detail is most helpful for you when reviewing architecture options?"
   - "How frequently would you like to review architectural decisions?"
   - "Are there specific aspects of the architecture you're most interested in being involved with?"

2. **Decision-Making Questions**:
   - "Who needs to be involved in key technology decisions?"
   - "What factors are most important to you when selecting technologies (cost, performance, familiarity, etc.)?"
   - "How quickly can we expect decisions on architectural questions that arise?"
   - "Are there any existing technology preferences or constraints we should be aware of?"

3. **Expectation-Setting Questions**:
   - "What does a successful architecture look like from your perspective?"
   - "What concerns do you have about the architectural process?"
   - "Are there any specific architectural qualities that are particularly important to you (scalability, performance, maintainability, etc.)?"
   - "What level of detail do you expect in architecture documentation?"

Remember that in Follow workflows, the architecture design process is collaborative and iterative. Plan for multiple touchpoints with the user and maintain meticulous documentation of all architectural decisions and their evolution over time.
